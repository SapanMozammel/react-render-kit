# Tailwind IntelliSense diagnostics (v4 codebase)

Two diagnostics are surfaced from **two sources of truth in parity**:

- **IDE**: `bradlc.vscode-tailwindcss` (Tailwind's official language server) — fires inline in VS Code.
- **CLI / CI**: `eslint-plugin-better-tailwindcss` (wired in `eslint.config.js`, generated by `.formatter/sync.js`) — fires via `pnpm run lint` and is auto-fixable via `pnpm run lint:fix`.

Both read the project's Tailwind v4 entry CSS (`src/styles/global.scss` `@theme`), so they agree on which utilities exist and which renames are canonical. The IDE setting (`tailwindCSS.classFunctions: ['cn', 'cva', 'tv', 'clsx']`) and the ESLint plugin's `callees` option are kept in sync via the formatter template — without that, neither tool fires inside `cn(...)`, which is where every className lives per the [`cn()` mandate](../architecture/component-patterns.md).

**The mapping tables below are reference documentation, not source of truth.** When Tailwind adds new renames, they ship via `pnpm update eslint-plugin-better-tailwindcss` — do NOT hand-maintain a mapping table in `/fix-tw-diagnostics` or this skill.

## 1. `suggestCanonicalClasses`

The class works (v3 backwards-compat in Tailwind v4), but a **canonical v4 form** exists. Two sub-cases:

### 1a. `!important` modifier — prefix → suffix

The canonical v4 placement is **suffix**, on the entire utility (after any variant prefix has done its work):

| v3 (legacy) | v4 (canonical) |
|---|---|
| `!h-9` | `h-9!` |
| `!leading-none` | `leading-none!` |
| `sm:!h-11` | `sm:h-11!` |
| `dark:!bg-success` | `dark:bg-success!` |
| `hover:!text-primary` | `hover:text-primary!` |
| `motion-reduce:!animate-none` | `motion-reduce:animate-none!` |
| `!px-[calc(theme(height.9)*21/44)]` | `px-[calc(theme(height.9)*21/44)]!` |

Same rule for `@apply` directives in SCSS:

```scss
/* v3 */
@apply font-hg text-3xl !leading-tight font-bold;

/* v4 */
@apply font-hg text-3xl leading-tight! font-bold;
```

Full reference: [`tailwind-v4-syntax`](./tailwind-v4-syntax.md).

### 1b. Renamed utilities — v3 alias → v4 canonical

These v3 names still resolve through Tailwind v4's deprecated-alias path but should be rewritten. **They are NOT emitted by the v4 generator**, so any post-build mangler won't find them in the CSS chunks and they ship un-mangled (or, if the alias gets removed in a future Tailwind, break outright).

| v3 alias | v4 canonical |
|---|---|
| `flex-shrink-0` | `shrink-0` |
| `flex-shrink` | `shrink` |
| `flex-grow-0` | `grow-0` |
| `flex-grow` | `grow` |
| `overflow-ellipsis` | `text-ellipsis` |
| `decoration-slice` | `box-decoration-slice` |
| `decoration-clone` | `box-decoration-clone` |
| `bg-gradient-to-{dir}` | `bg-linear-to-{dir}` (renamed in v4 to make room for `bg-conic-*` / `bg-radial-*`; covers all 8 directions: `t`, `tr`, `r`, `br`, `b`, `bl`, `l`, `tl`. Arbitrary-angle form `bg-gradient-to-[45deg]` → `bg-linear-[45deg]`. Color stops `from-*`, `via-*`, `to-*` stay unchanged.) |
| `bg-opacity-{n}` | `bg-{color}/{n}` (e.g. `bg-primary/50`) |
| `text-opacity-{n}` | `text-{color}/{n}` |
| `border-opacity-{n}` | `border-{color}/{n}` |
| `ring-opacity-{n}` | `ring-{color}/{n}` |
| `divide-opacity-{n}` | `divide-{color}/{n}` |
| `placeholder-opacity-{n}` | `placeholder-{color}/{n}` |

The opacity migrations require **pairing** with the matching color utility (the `/N` syntax is "color slash opacity"). Without a color companion, drop the opacity utility entirely — `bg-opacity-50` alone meant nothing without a `bg-{color}`.

### 1c. Arbitrary-property hint → v4 named utility

In Tailwind v3, several CSS properties required the **arbitrary-property hint** form to set inline — `prefix-[css-property:value]`. Tailwind v4 added named utilities for many of these, and the IntelliSense extension flags the v3 form as a canonical-class candidate.

The general transform: `prefix-[css-prop:VALUE]` → `prefix-css-prop-[VALUE]`. The CSS property name moves out of the bracket and becomes part of the utility name.

| v3 form | v4 canonical | CSS prop |
|---|---|---|
| `bg-[size:56px_56px]` | `bg-size-[56px_56px]` | `background-size` |
| `bg-[position:center_top]` | `bg-position-[center_top]` | `background-position` |
| `mask-[size:32px]` | `mask-size-[32px]` | `mask-size` |
| `mask-[position:center]` | `mask-position-[center]` | `mask-position` |
| `mask-[image:url(...)]` | `mask-image-[url(...)]` | `mask-image` |
| `[field-sizing:content]` | `field-sizing-content` | `field-sizing` (no prefix in v3; named utility in v4) |
| `[field-sizing:fixed]` | `field-sizing-fixed` | `field-sizing` |
| `[color-scheme:light]` | `color-scheme-light` | `color-scheme` |
| `[color-scheme:dark]` | `color-scheme-dark` | `color-scheme` |
| `[color-scheme:light_dark]` | `color-scheme-light-dark` | `color-scheme` |

**Caveats:**

- Not every `[css-prop:value]` form has a v4 named utility. If `/fix-tw-diagnostics` finds a `prefix-[css-prop:...]` pattern that isn't in the table above, it reports it as a candidate for manual review rather than auto-rewriting (no v4 named utility may exist).
- **Bare arbitrary properties** (`[--my-var:5]` — CSS custom properties without a prefix) are NOT migrated. Custom properties have no named utility; the bracket form is canonical.
- **Bare arbitrary values** (`bg-[url(...)]`, `text-[12px]`, `min-h-[calc(100vh-4rem)]`) are NOT migrated — there's no `css-prop:` inside the brackets, so the pattern doesn't match. The bracket form remains canonical for arbitrary values.

### How to fix

- **Single utility:** move the `!` to the end (1a), replace the alias with the canonical name (1b), or rewrite the arbitrary-property hint as the named utility (1c). Leave any variant prefix in place.
- **Whole-codebase sweep:** invoke `/fix-tw-diagnostics`, or just run `pnpm run lint:fix`. Both ultimately invoke the same ESLint rules:
  - `better-tailwindcss/enforce-consistent-important-position` — 1a
  - `better-tailwindcss/no-deprecated-classes` — 1b
  - `better-tailwindcss/enforce-canonical-classes` — 1c (and adjacent shorthand merges like `h-full w-full` → `size-full`)
  All three are autofixable; the rules read the project's Tailwind v4 entry CSS, so any rename the v4 generator knows about flows through automatically — there is no hand-maintained mapping table.
- **Full v3 → v4 migration (config + tokens + syntax):** `pnpm dlx @tailwindcss/upgrade` — broader scope; use when also migrating theme tokens. For a syntax-only sweep, the lint rules are the right tool.

## 2. `cssConflict`

Two utility classes in the same className string target the same CSS property; only the **rightmost** wins (Prettier's class sorter keeps both, so the conflict is invisible at the source level but live at runtime).

### Common conflict patterns

| Pattern | Why it conflicts | Likely intent |
|---|---|---|
| `p-4 px-2` | `p-*` sets all four sides; `px-2` overrides left/right → final = `p-4` top/bottom + `px-2` left/right | Usually intentional — keep both, **dismiss diagnostic** |
| `p-4 p-6` | Same property, two values — second wins | Pick one; delete the other |
| `text-red-500 text-blue-500` | Two `color` values | Pick one |
| `text-sm text-base` | Two `font-size` values | Pick one |
| `font-bold font-normal` | Two `font-weight` values | Pick one |
| `flex inline-block` | Two `display` values | Pick one |
| `rounded-full rounded-lg` | Two `border-radius` values | Pick one |
| `absolute relative` | Two `position` values | Pick one |
| `w-4 w-8` | Two `width` values | Pick one |
| `mt-4 my-2` | `my-*` overrides `mt-*` and `mb-*` | If `mt` was the intent, drop `my`; if `my` was the intent, drop `mt` |
| `container w-full` | `container` already sets `width: 100%` (+ responsive `max-width`) | Drop `w-full` — `container` is the stronger utility. If the goal was to *escape* container's responsive max-widths, use a non-container parent + `w-full` instead |
| `size-4 w-6` / `size-4 h-6` | `size-*` sets both `width` and `height`; the explicit `w-*`/`h-*` overrides one axis | Asymmetric sizing — usually intentional; dismiss. If symmetric, drop the duplicate |
| `inset-0 top-4` | `inset-*` sets all four sides; the side-specific utility overrides one | Usually intentional — dismiss |
| `rounded rounded-md` | Both set `border-radius` | Pick one |
| `border border-2` | Both set `border-width` | Pick one |

**Cross-family conflicts** (different utility prefixes, same CSS property) are the long tail — the IDE Problems panel (Tailwind CSS IntelliSense) catches these via its complete utility→CSS-property map. The `/fix-tw-diagnostics` heuristic only catches the high-frequency cases above. **Use the IDE as the source of truth for `cssConflict`; the command's Step 3 sweep is best-effort, not exhaustive.**

### Project-specific *non*-conflicts (do NOT flag)

The diagnostic occasionally flags pairs that are **intentional**:

- **`bg-light dark:bg-slate-900`** — dark-mode pair. Variants make these distinct utilities; the IDE diagnostic should not flag the pair, but if it does, dismiss.
- **`p-4 px-2`** (shorthand + axis override) — used in places where padding is uneven; keep both, dismiss the diagnostic at the call site.
- **`text-primary dark:text-success`** and other `light dark:dark` color pairs — same as above; variant gives each its own slot.

### How to fix

1. Read the line in context — figure out which utility expresses the intent.
2. Delete the loser.
3. If both look intentional and one carries a variant (`dark:`, `sm:`, `hover:`), they're not actually conflicting — confirm in the diff that the IDE underline goes away after the fix.

`cssConflict` cannot be auto-fixed safely — the resolution requires intent inference. `/fix-tw-diagnostics` will **report** conflicts grouped by file/line and leave the rewrite to the author.

## 3. How this interacts with the rest of the toolchain

| Tool | Role |
|---|---|
| `prettier-plugin-tailwindcss` | Sorts classes inside `cn()`/`className=""` left-to-right by Tailwind's recommended order. Does NOT rewrite v3 → v4 syntax. Does NOT merge conflicts. |
| `bradlc.vscode-tailwindcss` extension (IDE) | Lints with `suggestCanonicalClasses` and `cssConflict` inline in VS Code. Requires `tailwindCSS.classFunctions: ['cn', 'cva', 'tv', 'clsx']` to fire inside `cn(...)` (configured in `.formatter/sync.js`). |
| `eslint-plugin-better-tailwindcss` (CLI / CI) | Same two diagnostics via `pnpm run lint`. Four rules wired: `enforce-consistent-important-position` (1a), `no-deprecated-classes` (1b), `enforce-canonical-classes` (1c), `no-conflicting-classes` (cssConflict, report-only). The `callees` option mirrors the IDE's `classFunctions`. Reads `src/styles/global.scss` `@theme` for the live utility map. |
| `/fix-tw-diagnostics` | Thin wrapper around `pnpm run lint:fix` + a `cssConflict` report pass. Used to also be grep-based with a hardcoded mapping table — no longer; the plugin is the source of truth. |
| `/format` | Runs the full formatter pipeline. Tailwind canonical fixes now ride along via `lint:fix`. |
| Post-build mangler (if configured) | Only mangles classes that appear **both** in the v4 generated CSS and in emitted JS/HTML — v3 alias names are silently dropped on the mangle, which is why 1b matters. |

## What NOT to change

- Plain JavaScript negation (`!isOpen`, `!ctx`, `!body.contains(...)`) — these are JS, not Tailwind classes. The grep stays inside quoted strings to avoid them.
- `!important` declarations in raw CSS (`color: red !important`) — that's CSS spec, untouched by either Tailwind version.
- Prose inside content strings — `data/content/blogs/*.ts` often discusses CSS / `!important` semantically; do not rewrite blog body text.
- Canonical color pairs (`bg-light dark:bg-slate-900`, `text-primary dark:text-success`) — these are NOT conflicts, the variant disambiguates them.
